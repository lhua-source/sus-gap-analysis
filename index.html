⚠ Critical Gaps Impacting Recommendation Quality

1. Job 3: Customer Profile Updates — NOT STARTED (NDS-205)

Confluence Spec: Nightly at 4 AM. Per customer: purchase history (total orders, LTV, avg order value, recency), preferences (top 5-10 items, favorite categories, never-ordered, diversity score), behavioral patterns (preferred hours, days, locations), recommendation engagement (acceptance rate, last accepted/rejected), segmentation (price sensitivity, segment, lifecycle, risk). Output: customer_profiles cached in Redis + persisted to DB.
Current Code: frequentcustomerid is label-encoded as a single integer. No profile computation, no Redis cache, no nightly job. frequentcustomers.parquet loaded but only used for the ID.
Impact: 0/15 customer features. Known customers get population-level recs identical to guests. All V3 personalization blocked. Dual-model architecture impossible.

2. Job 5b: Dual Model Training — Only Guest Model Exists (NDS-208)

Confluence Spec: Dual Models (Recommended): Guest Model (45 features) + Personalized Model (60 features with 15 customer features including order_count, lifetime_value, avg_order_value, days_since_last, favorite_items, favorite_categories, never_ordered_items, item_diversity_score, price_sensitivity, acceptance_rate, preferred_hours, preferred_days).
Current Code: Single XGBoostRecsysTrainer trains one binary model on all orders. No model split by guest vs known. No 15 customer features.
Impact: No personalized model for known customers. Model cannot learn customer-specific patterns.

3. Category Hierarchies Not Materialized (NDS-211)

Confluence Spec (Job 4): "Build category hierarchies with Complementarity rules." Jira: Two Cosmos tables (Rules + Metadata), weekly builder job. Rules: CategoryId, RuleType, CategoryList[], Strengths[], Soft_limit. Metadata: TotalItems, price stats, PriceBuckets, ItemIDs[], RecommendCategories[], UpsellPath[], NeedsCategory[].
Current Code: Complement scores hardcoded in cpu_training.py calculate_category_complement_scores() (~100 lines). Category limits scattered across 3 config objects with different values. No Cosmos tables, no per-org customization.
Impact: Rules cannot be tuned per brand without code changes. Bojangles pairings not captured. Inconsistent limits across configs.

4. Item Weather Tags Missing (NDS-237)

Confluence Spec (Job 4): Extract item attributes including dietary tags. NDS-237: Per-item weather_tags: ['hot_weather', 'warm_weather'], temperature_affinity. Used in Training AND Inference to filter weather-inappropriate items.
Current Code: Weather features at order-level only: weather_temperature_c, weather_comfort_score. No per-item weather affinity.
Impact: On a 95°F Waxhaw day, iced tea and hot coffee get identical weather signals. No item-level weather awareness.

5. Cold Start (LLM) — Not Integrated in Pipeline

What NDS-208 Branch Gets Right (Not Yet Deployed)

The nds-208 branch has the right architecture but the binary training pipeline (training_pipeline.py) hasn't been merged to develop or used to produce the deployed model:

  • Binary classification (P(accept|cart,candidate)) instead of multiclass (P(item=j|features)) — directly models cart-candidate relevance
  • Leave-one-out training with negative sampling — creates proper binary labels
  • Complement scoring at inference — boosts complementary categories, penalizes same-category
  • Gap-Aware MMR ranking — ensures drink/side/dessert diversity in final set
  • CandidateFilter with 5-stage business rules — reduces candidate pool before scoring
  • item_feature_index would store per-item features for binary candidate scoring

Action Required: Train a binary model using the nds-208 pipeline, save with model_type="binary" and populated item_feature_index, then deploy.
